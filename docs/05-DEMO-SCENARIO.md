# MeetingMod - 해커톤 데모 시나리오

## 1. 데모 개요

### 목표
- **핵심 차별점** 강조: 실시간 회의 개입 (기존 솔루션과의 차별화)
- **Multi-Agent 아키텍처** 시연: OpenAI Agents SDK 활용
- **비즈니스 가치** 전달: 기업 회의 문화 개선

### 데모 환경
- **실행 환경**: localhost
- **오디오 입력**: 맥북 마이크 → Whisper STT
- **개입 출력**: 경고음 + Toast 메시지

### 데모 시간
- 예상 발표 시간: **5분**
- 실제 시연: **3분**
- Q&A 대비: **2분**

## 4. Q&A 예상 질문 및 답변

### Q1: "실시간 개입이 회의 흐름을 방해하지 않나요?"

```
A: 좋은 질문입니다.

첫째, 개입 타이밍을 조절합니다.
발화자가 말을 멈춘 후에만 개입해서, 말 끊지 않습니다.

둘째, 개입 빈도를 제한합니다.
연속 개입 사이에 최소 2분 간격을 두고,
정말 필요한 경우에만 개입합니다.
```

### Q2: "화자 분리는 어떻게 하나요?"

```
A: 참석자 목록을 기반으로 AI가 자동 분리합니다.
발화 내용, 역할, 이전 발화 컨텍스트를 종합해서
GPT-4o가 화자를 추론합니다.
```

### Q3: "회의 원칙 위반은 어떻게 감지하나요?"

```
A: LLM이 직접 판단합니다.
설정된 원칙 내용을 컨텍스트로 전달하고,
발화 내용이 원칙과 충돌하는지 분석합니다.
예를 들어 '수평적 의사결정' 원칙이 있는데
일방적 결정 발언이 나오면 감지합니다.
```

### Q4: "데이터는 어디에 저장되나요?"

```
A: 모든 데이터는 로컬 Markdown 파일로 저장됩니다.
회의별로 디렉토리가 생성되고,
녹취록, 개입 기록, 요약 등이 .md 파일로 남습니다.
버전 관리나 검색이 쉽고, 가독성도 좋습니다.
```

### Q5: "Agent가 오판하면 어떻게 하나요?"

```
A: 모든 개입에는 '무시' 버튼이 있습니다.
오판 시 사용자가 dismiss 할 수 있고,
해당 피드백은 향후 정확도 개선에 활용됩니다.
```

## 5. 백업 플랜

### 시나리오 A: STT가 동작하지 않을 때
- 시뮬레이션 모드로 전환
- 미리 준비된 스크립트를 자동 재생
- "데모를 위해 시뮬레이션 모드로 진행합니다"

### 시나리오 B: Agent 응답이 느릴 때
- 사전 준비된 개입 메시지를 수동 트리거
- 키보드 단축키로 Toast 강제 출력
- "Agent가 분석 중인데, 결과를 미리 보여드리겠습니다"

### 시나리오 C: 화자 분리가 부정확할 때
- UI에서 화자 수동 선택 버튼 활용
- "실제 환경에서는 여러 발화를 통해 정확도가 높아집니다"

### 시나리오 D: 전체 시스템 문제
- 로컬 녹화 영상으로 대체
- "라이브 데모 대신 녹화 영상을 보여드리겠습니다"

## 7. 심사 기준 대응

| 평가 항목 | MeetingMod 대응 |
|----------|----------------|
| **기술 완성도** | 실시간 STT → AI 화자 분리 → Agent 분석 → 경고음 + Toast 파이프라인 |
| **Multi-Agent 활용** | Prep/Moderator/Review 3개 Agent 협업 + OpenAI Agents SDK |
| **비즈니스 가치** | 기업 회의 문화 개선, 회의 원칙 코칭, 시간 절약 |
| **차별성** | "기록에서 개선으로" - 실시간 개입 + 원칙 기반 피드백 |
| **확장성** | Markdown 기반 원칙 편집, 다국어 확장 가능, 화상회의 연동 로드맵 |
